PLAIN ENGLISH

Opening brace — this is where the profile object starts.

Name field starts blank — we'll fill this in from the user's info.

Target role is set to "AI 应用开发实习生" by default — the role we're training for.

topic_mastery — a dictionary that maps each topic to your skill level (1 to 5) and a notes field. Starts empty for new users.

As you answer questions, the system fills this in automatically.

weak_points — a list of your weaknesses. Each entry records: what the issue is, which topic it belongs to, when it was first seen, when last seen, and how many times it appeared.

strong_points — a list of your strengths. Simpler than weak_points: just the strength description, the topic, and when it was first spotted.

communication — tracks how you express yourself, including your answer style, your speaking habits, and personalized improvement suggestions.

style: e.g. "回答偏短，缺少具体例子" — a direct summary of your pattern.

habits: e.g. ["紧张时语速加快", "喜欢用类比解释"] — your verbal tendencies.

suggestions: e.g. ["多用 STAR 法描述项目"] — coaching tips generated for you.

thinking_patterns — how your brain approaches problems.

strengths: e.g. ["能用类比解释抽象概念"] — things you do well in terms of reasoning structure.

gaps: e.g. ["对比类问题缺乏结构"] — types of questions where your thinking lacks structure.

stats — your training scorecard.

total_sessions: starts at 0, counts up with every training session.

avg_score: your running average across all sessions.

score_history: a list of every past session's details — date, mode, topic, and average score.

Closing brace — end of the profile object.

SM-2 算法 — 面试复习调度

1987年的记忆科学，如何算出"下次复习的最佳时机"

3-1 为什么你总在同一道题上跌倒？

答对了一道题，下次还出吗？大多数系统随机抽，结果你总是忘的那题最久才来。

🎲

随机出题

每隔固定时间随机抽题，不管你会不会。

周一 忘了
……
周四 又忘了！
……
下周一 还是忘了……
😫

每次都像重新学，永远在同一个坑里跌倒。

🧠

SM-2 调度

对了拉长间隔，错了一天后重考——动态适应你的记忆。

周一 对了 ✓
→
周二 对了 ✓
→
周五 对了 ✓
→
再过14天 巩固 ✓
💪

在你快忘的时候出现，每次复习都恰到好处。

⚡
SM-2 的核心逻辑

答对了？→ 拉长间隔，给你时间巩固。
答错了？→ 重置回1天，明天就回来救你。

3-2 SM-2 的两个核心数字

SM-2 算法只用两个数字，就能量化你的记忆状态。

📅

interval_days

间隔天数——"多久之后再考你一次"。从1天起步，每次答对就翻倍：1天 → 3天 → 7天 → 14天……

答对：间隔 = 上次间隔 × 难度系数
⚙️

ease_factor

难度系数——"这道题到底有多难"。初始默认2.5，每次答题后根据表现微调，最低不低于1.3。

答难→高分 → 系数变大 → 间隔涨得更快
📊
state 是什么？

state 就是这两个数字加上一个 repetitions（连续答对次数）的组合，每次答题后 SM-2 用这三个数算出下一轮复习时间。这组数据存在 JSON 里，叫 sr_state。

3-3 逐行拆解：sm2_update 函数

这是 TechSpar 后端的实际代码，逐行翻译成大白话。

CODE 
def sm2_update(sr_state: dict, score_0_10: float) -> dict:
    quality = min(5, int(score_0_10 / 2))
    # 0-10分 → SM-2质量0-5
    ef = sr_state.get("ease_factor", 2.5)
    # 难度系数，最小1.3
    reps = sr_state.get("repetitions", 0)

    if quality >= 3:  # 通过
        if reps == 0: interval = 1
        elif reps == 1: interval = 3
        else: interval = int(sr_state.get("interval_days", 1) * ef)
        reps += 1
    else:  # 失败，重置
        interval = 1
        reps = 0

    ef = max(1.3, ef + (0.1 - (5 - quality) * (0.08 + (5 - quality) * 0.02)))
    return {
        "interval_days": interval,
        "ease_factor": round(ef, 2),
        "repetitions": reps,
        "next_review": (date.today() + timedelta(days=interval)).isoformat(),
        "last_score": score_0_10,
    }
PLAIN ENGLISH

Define a function that takes the current memory state of a question and a score (0-10), then returns the updated state for the next review session.

Convert the 0-10 score into SM-2's internal "quality" scale (0-5) by dividing by 2 and capping at 5. A 10/10 score becomes quality=5.

-

Look up the ease_factor from the current state. If there's no record yet, default to 2.5 — the standard starting difficulty for any new question.

-

Also look up how many times in a row this question has been answered correctly. Start at 0 if it's a fresh question.

-

If the quality is 3 or higher (answered correctly enough): figure out the next review interval.

First time correct → review again in 1 day.

Second time correct → now 3 days.

Third+ time → multiply the previous interval by the ease_factor. This is the compounding magic: each success stretches the interval further.

Increment the consecutive-correct counter by 1.

If the quality is below 3 (failed or shaky): reset the interval to 1 day and start the counter over from zero.

Interval goes back to 1 day.

Counter resets to 0.

-

After all this, update the ease_factor using a formula based on how well the question was answered. Never let it drop below 1.3.

-

Return all four updated values so the system knows when to bring this question back.

interval_days — how many days until the next review.

ease_factor — updated difficulty coefficient (rounded to 2 decimal places).

repetitions — how many times answered correctly in a row now.

next_review — the actual calendar date for the next review, calculated as today + interval_days.

last_score — the score you just gave, stored for the system's records.

关键洞察

SM-2 用两个简单的规则模拟了记忆的衰减规律：答对说明记忆还牢固，多给几天让它自然淡忘再回来考；答错说明记忆已经不牢，就不要硬拖时间，立刻回来救你。

SM-2 算法

模块 4 · 屏幕 2 / 5

每个阶段的具体任务

LangGraph 的状态机把面试拆成 5 个节点， 每个节点只做一件事，然后通过边决定下一步去哪。

1

GREETING · 开场寒暄

"你好，欢迎参加面试，今天准备得怎么样？"

👋

AI 问候候选人，确认设备正常、候选人状态放松。目的是消除紧张感，建立自然的对话氛围。这个阶段通常只需要 1–2 条消息即进入下一阶段。

2

SELF_INTRO · 自我介绍

"请简单介绍一下你自己，以及你最近的项目经历。"

🎤

候选人介绍自己的工作背景和技能方向。AI 从中提取关键词，用于后续技术问题的个性化调整。如果候选人回答太简短，AI 会追问一句补充说明。

3

TECHNICAL · 技术问题

"聊聊你对分布式系统的理解，以及你用过的相关技术。"

💻

核心能力考察阶段。AI 从编程语言、算法、系统设计等角度出题，并根据回答深入追问。关键机制：每题回答后 AI 会评估回答质量，决定继续追问还是换话题。当该阶段问题数达到 HARD_MAX_PER_PHASE = 10 时，无论是否充分，都强制切换到下一阶段。

4

PROJECT_DEEP_DIVE · 项目深挖

"你简历里提到这个项目，请具体说说用了什么架构？"

🔍

基于简历上下文，深挖项目细节：技术选型原因、最大的技术挑战、如何解决冲突、团队协作方式等。这个阶段会追问多轮，直到问题深度足够或达到 HARD_MAX。

5

REVERSE_QA · 反问环节

"你有什么问题想问我们吗？"

❓

候选人向 AI 提问。AI 切换为"回答者"角色，诚实回复关于团队、技术栈、晋升通道等问题。候选人没有更多问题或达到 HARD_MAX 时，is_finished = True，状态机退出。

模块 4 · 屏幕 3 / 5

代码直译：阶段顺序与状态定义

下面两段代码是 LangGraph 面试图的骨架。 左侧是真实代码，右侧是逐行英文翻译。

CODE 
# 阶段执行顺序 — 按这个列表一个接一个跑
PHASE_ORDER = [
    InterviewPhase.GREETING.value,          # 开场寒暄
    InterviewPhase.SELF_INTRO.value,       # 自我介绍
    InterviewPhase.TECHNICAL.value,         # 技术问题
    InterviewPhase.PROJECT_DEEP_DIVE.value, # 项目深挖
    InterviewPhase.REVERSE_QA.value,        # 反问环节
]
HARD_MAX_PER_PHASE = 10  # 每阶段最多10题，防止无限循环
PLAIN ENGLISH

A comment telling developers this list controls the execution order — one phase runs after another.

Define a constant named PHASE_ORDER and assign it a list of values.

Get the string value ("greeting") from the GREETING option in the InterviewPhase enum, and put it first in the list.

Get the string value ("self_intro") from the SELF_INTRO option and put it second.

Get the string value ("technical") and put it third.

Get the string value ("project_deep_dive") and put it fourth.

Get the string value ("reverse_qa") and put it last — the reverse-questioning phase.

End of the list.

Set HARD_MAX_PER_PHASE to 10 — a safety limit so the AI never asks more than 10 questions in any single phase. Prevents infinite loops.

CODE 
class ResumeInterviewState(TypedDict):
    messages: list[BaseMessage]      # 对话历史
    resume_context: str             # 简历上下文
    phase: str                  # 当前阶段
    questions_asked: list[str]   # 已问问题
    phase_question_count: int # 当前阶段已问题数
    is_finished: bool           # 是否结束
    eval_history: list[dict]     # 每题评估历史
PLAIN ENGLISH

Define a class named ResumeInterviewState that inherits from TypedDict — this is a typed dictionary where each field has a known type. Think of it as a structured form that the state machine fills in.

A field called "messages" holds a list of BaseMessage objects — every back-and-forth in the conversation gets recorded here so the AI remembers what was said.

A field called "resume_context" holds the candidate's resume as a plain text string — the AI reads this to know what projects and skills to ask about.

A field called "phase" holds a string like "greeting" or "technical" — this tells the state machine which phase we are currently in.

A field called "questions_asked" holds a list of strings — every question the AI has already asked, so it doesn't ask the exact same thing twice.

A field called "phase_question_count" holds an integer — counts how many questions have been asked in the current phase. Used to check if HARD_MAX has been reached.

A field called "is_finished" holds a True/False value — when this becomes True, the state machine knows to exit and end the interview.

A field called "eval_history" holds a list of dictionaries — after each question, the AI records its evaluation of the answer here, like a grading log.

💡
关键洞察

所有 6 个字段加在一起，就是 LangGraph 的状态。 每当 AI 问一个问题，状态就更新一次：phase_question_count + 1， 直到 phase_question_count == HARD_MAX 时，触发阶段切换。 这就是"状态机"的工作方式——状态驱动行为。

React（前端）

⚡
FastAPI（后端）

我是中枢神经——FastAPI！用户发请求，先到我这儿。我负责：接收请求 → 调用 LangGraph → 把 AI 回复流式推回给 React。我是指挥，不亲自「演奏」（不生成答案），但协调所有人。

🧠
LangGraph（流程引擎）

我是流程编排引擎！面试不是一段直线——有时候要追问，有时候要给提示，有时候要切换到薄弱点强化模式。我用「状态机」把这些分支管理得井井有条。每个状态转换都是我来驱动的。

📋
Profile.json（记忆）

我负责记住一切！用户的薄弱点、SM-2 参数、上次面试答得最差的题目……FastAPI 开始处理请求前，会先来我这查档案；结束后，会把新的评估结果写回给我。没有我，每次训练都是新人。

📅
SM-2（复习调度器）

我是「遗忘规律」的执行者！根据间隔重复算法，我会计算出用户下次应该在什么时候复习哪道题。用户可能在 3 天后来，也可能 10 天后——我不会让他们在已经掌握的题目上浪费时间。

?

关键理解

这五个组件互相配合，任何一个出问题都会影响整体体验。定位问题的第一步，就是判断「这是哪一层的问题」。

模块 6

从代码看懂架构 — docker-compose.yml

这个配置文件用一行话解释就是：「有哪些服务，各自运行在哪个端口，如何通信」。

CODE 
# docker-compose.yml — TechSpar 服务架构
Frontend:
  image: react19+vite+tailwind
  ports: ":5173"
  # 用户看到的所有界面

Backend:
  image: fastapi+langchain+langgraph
  ports: ":8000"
  # 所有业务逻辑

SQLite:
  image: local-persist
  # 本地持久化存储

JWT:
  auth: enabled
  # 身份认证

LLM:
  api: openai-compatible
  # 可接入任意大语言模型
PLAIN ENGLISH

这是整个 TechSpar 的服务清单配置文件——它告诉 Docker（容器平台）应该启动哪些服务。

Frontend（前端）用 React 19 + Vite + Tailwind CSS 构建，运行在电脑的 5173 端口。用户打开浏览器访问的就是这个。

Backend（后端）用 FastAPI + LangChain + LangGraph 构建，运行在 8000 端口。它接收前端的请求，调用 AI，处理数据。

SQLite 是一个轻量级本地数据库，所有结构化数据（用户答案、历史记录）都存在这里。

JWT 是身份认证机制——确保每个请求都来自合法登录用户，防止陌生人访问他人数据。

LLM（Large Language Model）是对接大语言模型的接口——只要是兼容 OpenAI 格式的 API 都能接入，比如 GPT-4、Claude、本地模型都可以。

三层架构的每个组件都可以独立升级或替换——比如把 LLM 从 GPT-4 换成 Claude，不需要动前端和数据库。这就是「解耦」的价值。

模块 6

为什么这样分层？

架构不是随意划分的，每一层都有它存在的理由。

⚛️

前端为什么「轻」？

界面需要快速响应、流畅体验，但不需要复杂的 AI 运算。把界面和智能分开，前端只负责「好看和交互」，AI 推理交给后端。

⚡

后端为什么「重」？

所有 AI 逻辑、数据处理、认证都在后端。集中在一处的好处是：安全（AI key 不泄露）、可控（可以在服务器上优化性能）。

🧠

LangGraph 的价值

不用 LangGraph 也可以写 AI 应用，但会陷入「一堆 if-else」的混乱。状态机让流程可视化、可调试、可扩展——加新功能只需要加节点和边。

📋

为什么是 JSON 文件？

对于用户量不大的个人工具来说，JSON 文件够用且简单。随时可以迁移到数据库，但一开始不需要承担数据库的运维成本。

💡
架构思维

判断「这个功能应该加在哪层」，是架构思维的核心。记住一个原则：交互逻辑 → 前端；业务逻辑 + AI → 后端；持久数据 → 数据库。遇到新需求，先问「这主要是改用户看到的东西，还是改数据处理的逻辑？」
