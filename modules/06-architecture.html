模块 6

组件总动员 — 5 个角色自我陈述

每个组件都有自己的「岗位职责」。让我们听听它们各自怎么说：

⚛️
React（前端）

大家好！我是用户看到的那一层——界面。我用 React 19 搭建，用 Vite 跑起来。用户点的每一个按钮、看到的每一条 AI 回复，都是我负责渲染的。

⚡
FastAPI（后端）

我是中枢神经——FastAPI！用户发请求，先到我这儿。我负责：接收请求 → 调用 LangGraph → 把 AI 回复流式推回给 React。我是指挥，不亲自「演奏」（不生成答案），但协调所有人。

🧠
LangGraph（流程引擎）

我是流程编排引擎！面试不是一段直线——有时候要追问，有时候要给提示，有时候要切换到薄弱点强化模式。我用「状态机」把这些分支管理得井井有条。每个状态转换都是我来驱动的。

📋
Profile.json（记忆）

我负责记住一切！用户的薄弱点、SM-2 参数、上次面试答得最差的题目……FastAPI 开始处理请求前，会先来我这查档案；结束后，会把新的评估结果写回给我。没有我，每次训练都是新人。

📅
SM-2（复习调度器）

我是「遗忘规律」的执行者！根据间隔重复算法，我会计算出用户下次应该在什么时候复习哪道题。用户可能在 3 天后来，也可能 10 天后——我不会让他们在已经掌握的题目上浪费时间。

?
💡
关键理解

这五个组件互相配合，任何一个出问题都会影响整体体验。定位问题的第一步，就是判断「这是哪一层的问题」。

模块 6

从代码看懂架构 — docker-compose.yml

这个配置文件用一行话解释就是：「有哪些服务，各自运行在哪个端口，如何通信」。

CODE 
# docker-compose.yml — TechSpar 服务架构
Frontend:
  image: react19+vite+tailwind
  ports: ":5173"
  # 用户看到的所有界面

Backend:
  image: fastapi+langchain+langgraph
  ports: ":8000"
  # 所有业务逻辑

SQLite:
  image: local-persist
  # 本地持久化存储

JWT:
  auth: enabled
  # 身份认证

LLM:
  api: openai-compatible
  # 可接入任意大语言模型
PLAIN ENGLISH

这是整个 TechSpar 的服务清单配置文件——它告诉 Docker（容器平台）应该启动哪些服务。

Frontend（前端）用 React 19 + Vite + Tailwind CSS 构建，运行在电脑的 5173 端口。用户打开浏览器访问的就是这个。

Backend（后端）用 FastAPI + LangChain + LangGraph 构建，运行在 8000 端口。它接收前端的请求，调用 AI，处理数据。

SQLite 是一个轻量级本地数据库，所有结构化数据（用户答案、历史记录）都存在这里。

JWT 是身份认证机制——确保每个请求都来自合法登录用户，防止陌生人访问他人数据。

LLM（Large Language Model）是对接大语言模型的接口——只要是兼容 OpenAI 格式的 API 都能接入，比如 GPT-4、Claude、本地模型都可以。

为什么分离很重要

三层架构的每个组件都可以独立升级或替换——比如把 LLM 从 GPT-4 换成 Claude，不需要动前端和数据库。这就是「解耦」的价值。

模块 6

为什么这样分层？

架构不是随意划分的，每一层都有它存在的理由。

⚛️

前端为什么「轻」？

界面需要快速响应、流畅体验，但不需要复杂的 AI 运算。把界面和智能分开，前端只负责「好看和交互」，AI 推理交给后端。

⚡

后端为什么「重」？

所有 AI 逻辑、数据处理、认证都在后端。集中在一处的好处是：安全（AI key 不泄露）、可控（可以在服务器上优化性能）。

🧠

LangGraph 的价值

不用 LangGraph 也可以写 AI 应用，但会陷入「一堆 if-else」的混乱。状态机让流程可视化、可调试、可扩展——加新功能只需要加节点和边。

📋

为什么是 JSON 文件？

对于用户量不大的个人工具来说，JSON 文件够用且简单。随时可以迁移到数据库，但一开始不需要承担数据库的运维成本。

💡
架构思维

判断「这个功能应该加在哪层」，是架构思维的核心。记住一个原则：交互逻辑 → 前端；业务逻辑 + AI → 后端；持久数据 → 数据库。遇到新需求，先问「这主要是改用户看到的东西，还是改数据处理的逻辑？」
